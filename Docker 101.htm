What is Docker?

 

A way to encapsulating everything that you need to run a application. Docker container put that all together into one nice package.  A practical standpoint it's just a way to package software so it can run on any hardware.  Similar to VM just lighter weight,

 

Three things must know

Docker files - docker file is a blueprint for building a docker image

images - a docker image is a template for running docker containers

containers - a container is just a running process

 

In our case, we have a node application we need to have a server that's running the same version of node and that has also installed these dependencies. It works on my machine but if someone else with a different machine tries to run it with a different version of node it might break.  The whole point of docker is to solve problems like this by reproducing environments. The developer who creates the software can define the environment.

 

With a docker file, any developer can use the docker file to rebuild the environment which is saved as an immutable snapshot known as an image. Images can be uploaded to the cloud in both public and private registries, then any developer or server that wants to run that software can pull the image down to create a container which is just a running process of that image.

You build a docker image by running thedocker build command.

 

$ docker build [OPTIONS] PATH | URL | -

 

$ docker build -t dockerdemo/demoapp:1.0 .

 

The repository name will be fireship/demoapp and the tag will be 1.0.

 

$ docker build -t 101dockerweb:latest .

Default

Description

Name and optionally a tag in the 'name:tag' format

Docker container

 

 docker run [OPTIONS] IMAGE [COMMAND] [ARG...]

 

$ Docker run -d -p 5050:8080 --name dockerdemo-web dockerdemo/demoapp:1.0

 

$ docker run -p 6050:5050 -p 6051:5051 -e ASPNETCORE_HTTPS_PORT=https://+:5051 -e ASPNETCORE_URLS=http://+:5050 101dockerweb

 

$ docker run -p 5050:80 --name 101DockerWeb -e ASPNETCORE_URLS=http://+:80 101dockerweb

 

Option

 

Name, shorthand

Default

Description

--publish , -p

 

Publish a container's port(s) to the host

--name

 

Assign a name to the container

--detach , -d

 

Run container in background and print container ID

-it

 

The -t and -i flags allocate a pseudo-tty and keep stdin open even if not attached. This will allow you to use the container like a traditional VM as long as the bash prompt is running.

-rm

 

Automatically remove the container when it exits

 

 

This binds port8080 of the container to port 5000 of the host machine.

 

When you stop the container any state ordata that you created inside of it willbe lost.

 

 

Docker Volume

 

 

 

If you want to share data across multiple containers, The preferred way to do that is withvolumes. A volume is just a dedicated folder on the host machine, and inside this folder a container can create files that can be remounted into future containers or multiple containers at the same time.

 

$ docker run -d -p 5000:8080 --name dockerdemo-web --mount source=shared-stuff,target=/stuff dockerdemo/demoapp:1.0

 

Option

 

Name, shorthand

Default

Description

--mount

 

Attach a filesystem mount to the container

--interactive , -i

 

Keep STDIN open even if not attached

--tty , -t

 

Allocate a pseudo-TTY

 

 

 

l The source of the mount. For bind mounts, this is the path to the file or directory on the Docker daemon host. May be specified as source or src.

l The destination takes as its value the path where the file or directory is mounted in the container. May be specified as destination, dst, or target.

 

Consider a case where you have a directory source and that when you build the source code, the artifacts are saved into another directory, source/target/. You want the artifacts to be available to the container at /stuff/, and you want the container to get access to a new build each time you build the source on your development host. How to Commit Changes to a Docker Image

 

When working with Docker images and containers, one of the basic features is committing changes to a Docker image. When you commit to changes, you essentially create a new image with an additional layer that modifies the base image layer.

 

Step 1: Pull a Docker Image

l sudo docker pull ubuntu

l sudo docker images

 

Step 2: Deploy the Container

l sudo docker run -it cf0f3ca922e0 bin/bash

 

Step 3: Modify the Container

l apt-get install nmap

l nmap --version

l exit

l sudo docker ps -a

 

Step 4: Commit Changes to Image

l sudo docker commit [CONTAINER_ID] [new_image_name]

l sudo docker commit deddd39fa163 ubuntu-nmap

l sudo docker images

 

But docker commit is ani-pattern.  Because you may break or forget something or lose the original script and have to start it all again. Better to use docker file instead.
