1)  initialize a git repository

 

> git init

 

2)  Add the change to staged

 

> git add .

 

3)  Remove the change from staged

 

> git restore --staged <filename>

 

4)  Rollback the change (if files in changed state)

 

> git restore <filename>

 

5)  Commit the change (-m = naming the change)

 

> git commit -m “starting project”

 

6)  Get the branch info

 

> git branch

 

7)  Get the log info 

 

> git log

 

Note: press q to exit log

 

8)  Get status of change

 

> git status

 

9)  Goto the previous committed

 

> git checkout <committed ID>

 

10)  Rollback to latest committed from  previous committed

 

> git checkout main

 

Note: old version is named “master”.

11)  Go back to latest committed

 

> git checkout -- .

 

12)  Create another branch for new feature

 

> git checkout -b <new branch name>

 

 

 

 

 

13)  Switch to master branch

 

> git checkout master

 

14)  Merge 2 branches together from master

 

> git merge <new branch name>

 

 

 

Let us take an example if we have a project with 3 commits on the master branch as commit 1,2,3 and feature branch commits as commit A and B. If we perform a git merge operation then commits A and B will be merged as commit 4 onto the master branch. 

 

 

 

15)  Rebase the branches together from master

 

> git rebase <new branch name>

 

Let us take an example if we have a project with 3 commits on the master branch as commit 1,2,3 and feature branch commits as commit A and B. If we perform a git rebase operation then the commits A and B will be rebased on to the master branch as commit 4 and 5 and there will be no logs of the feature branches. This is depicted in Fig 12.

 

 

 

Git Merge Vs Git Rebase:

Merge

Rebase

Git merge is a command that allows you to merge branches from Git.

Git rebase is a command that allows developers to integrate changes from one branch to another.

In Git Merge logs will be showing the complete history of the merging of commits.

Logs are linear in Git rebase as the commits are rebased 

All the commits on the feature branch will be combined as a single commit in the master branch.

All the commits will be rebased and the same number of commits will be added to the master branch.

Git Merge is used when the target branch is shared branch

Git Rebase should be used when the target branch is private branch

 

 

 

In git merge, looking at the end diagram, one can make out Commit A and B came from the feature branch. However, in the git-rebase diagram, it is difficult to understand where commits A and B came from. Therefore, we do git merge when we want our team to understand logs in a way where they can identify where each commit is coming from. We use Git Rebase when the logs of the repository will not be referred by anyone else. To summarise, we can use Git Rebase, when we are working on branches, which cannot be seen by other developers. And we use Git Merge when the target and source branch can be viewed by other developers.

simply choose “current change”, “incoming change”, “both”, or just modify your code and re-commit again.

 

 

 

 

 

 

18)  Reset the committed history

 

> git reset --hard  <-committed ID->

 

 

 

 

 

19)  Create the remote repository in Github

 

> git remote add origin https://github.com/ArchieFan/Git.git

 

Note: “origin” is the name of the remote repository.

 

20)  Check the remote repository 

 

> git remote

or

> git remote -v

 

21)  Push the local master branch to remote origin

 

> git push -u origin master

 

 

22)  Check the connection between local and remote branch

 

> git branch -r

 

23)  Clone the github image to local

 

> git clone https://github.com/ArchieFan/Git.git

 

 

24)  Download the github image to local origin/master - git pull (or git fetch + git merge) 

 

> git fetch

 

Note: Git Fetch is the command that tells the local repository that there are changes available in the remote repository without bringing the changes into the local repository.  

 

25)  Merge the local origin/master to local folder

 

> git merge origin/master

 

 

26)  directly pull all the file from remote branch to local folder 

Note: git pull (or git fetch + git merge)

 

> git pull

 

Note: Git Pull on the other hand brings the copy of the remote directory changes into the local repository.

 

27)  delete the remote repository 

 

> git remote rm origin
