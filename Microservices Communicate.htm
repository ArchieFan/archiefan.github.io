Brokerless Design

 

 

where all services talk to each other directly

 

 

 

 

back-end team provides an API or a contract to the front-end team who can then invoke these apis whenever needed. it's simple it's fast and it's super easy to get started with.

 

Brokerless - Read Path

 

 

 

 

 

aggregator service which would get the initial request it would fire appropriate queries on the user and email service to get the information we need.  stitch the data together and then send back a consolidated response.

 

 

 

graphql is a great tool as an aggregator

 

 

Brokerless - Write Path

 

 

 

 

 

 

 

If i want to update my dashboard service and a bi tool as well, we simply make the user service call their apis one by one, right? No, if i was your user service i would be pissed.

 

 

 

 

Brokerless Design - Problems

 

 

 

if the user service breaks for some reason tomorrow a lot of services would be affected

 

 

 

 

tight coupling problem is that instead of building a microservices architecture like we intended to. We end up building a tightly coupled distributed monolith. A distributed monolith is worse than a traditional monolith because all your api calls here are happening over the network.

 

if your services are depending on each other a lot you may want to merge them into one

 

Brokered Design

 

 

where all communications flow through a messaging broker

 

 

 

a pub sub broker is basically a communication system where a sender publishes messages on a topic on one side and the interested participants can consume or subscribe for those messages on the other.

 

Advantages - 1. Control over the API

 

 

 

The user service doesn't really need to track the apis of the email service or any other service. it doesn't care about.  instead it can now describe the topic and the format of the message it will publish and the interested participants can subscribe to that.

 

Advantages - 2. Asynchronous Processing

 

 

second we can now shift to asynchronous processing without having to make the consumers themselves asynchronous. What i mean by that is that the user service can return back a response as soon as it publishes a message. It doesn't really have to wait for the email service to be done. The messaging broker would make sure that automatic retries happen in case of failures.

 

Advantages - 3. Decoupled communication

 

 

 

 the user service need not be updated when a fourth service comes into the picture
