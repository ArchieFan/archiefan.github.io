A data-tier application (DAC) is a logical database entity that defines all of the SQL Server objects - such as tables, views, and instance objects, including logins - associated with a user's database. A DAC is a self-contained unit of the entire database model and is portable in an artifact known as a DAC package, or .dacpac.

If you're not using YAML, you can set up your CI/CD pipeline for SQL Server in Azure DevOps using the classic editor with the following steps:

  1. Create a New Pipeline:
    • Navigate to Pipelines > Pipelines in your Azure DevOps project.
    • Click on "New Pipeline".
    • Select your repository (e.g., Azure Repos Git, GitHub, etc.).
    • Choose "Use the classic editor" at the bottom of the setup page.
  2. Configure the Pipeline:
    • Select a template: You can start with an empty job or a predefined template that suits your needs.
  3. Add Build Steps:
    • Build the Database Project:
      • Add a "Visual Studio Build" task if you have a SQL Server Database project.
      • Configure the task to build your solution.
  4. Publish Build Artifacts:
    • Add a "Publish Build Artifacts" task to store the build output (e.g., DACPAC file) which will be used in the release pipeline.
  5. Create a Release Pipeline:
    • Navigate to Pipelines > Releases.
    • Click on "New pipeline".
    • Select a template or start with an empty job.
  6. Add Artifacts:
    • Link the build artifact from the build pipeline.
  7. Add Deployment Steps:
    • Add a "Azure SQL Database Deployment" task.
    • Configure the task with:
      • Azure subscription: Link your Azure subscription.
      • Azure SQL Server name: Your SQL server name.
      • Database name: Your database name.
      • Login and password: Credentials for your SQL server.
      • Package or file: The path to the DACPAC file from the build artifact.
  8. Save and Queue the Pipeline:
    • Save your pipeline.
    • Queue the pipeline to run the build and deployment process.

By following these steps, you can set up a CI/CD pipeline for SQL Server without writing YAML, using the Azure DevOps classic editor to configure the necessary tasks and actions visually.
